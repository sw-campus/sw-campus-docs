## 1. 배경 및 목적

### 1.1 배경

SW Campus 관리자가 플랫폼에 가입한 회원 정보를 조회하고 관리할 수 있는 기능이 필요합니다.

### 1.2 목적

- 관리자가 회원 목록을 조회하고 검색할 수 있도록 지원
- 회원 관리 업무 효율화
- 향후 회원 관리 기능(승인, 정지 등) 확장을 위한 기반 마련

---

## 2. 사용자 정의

### 2.1 사용자 유형 및 권한

| 사용자 유형 | Role | 회원 목록 조회 | 회원 검색 |
|------------|------|:-------------:|:---------:|
| 일반 사용자 | `USER` | - | - |
| 기관 담당자 | `ORGANIZATION` | - | - |
| 관리자 | `ADMIN` | O | O |

### 2.2 회원 조회 흐름

```
[ADMIN]
로그인 → 관리자 대시보드 → 회원 관리 → 회원 목록 조회/검색
```

---

## 3. 기능 요구사항

### 3.1 회원 목록 조회

| 구분 | 사용자 스토리 |
|------|-------------|
| US-01 | ADMIN은 전체 회원 목록을 페이지네이션으로 조회할 수 있다 |
| US-02 | ADMIN은 회원 목록을 가입일 기준 최신순으로 정렬하여 조회한다 |

### 3.2 회원 검색

| 구분 | 사용자 스토리 |
|------|-------------|
| US-03 | ADMIN은 이름으로 회원을 검색할 수 있다 |
| US-04 | ADMIN은 닉네임으로 회원을 검색할 수 있다 |
| US-05 | ADMIN은 이메일로 회원을 검색할 수 있다 |
| US-06 | 검색어는 이름, 닉네임, 이메일 중 하나라도 일치하면 결과에 포함된다 (OR 조건) |

### 3.3 조회 정보

| 구분 | 사용자 스토리 |
|------|-------------|
| US-07 | ADMIN은 회원의 ID, 이메일, 이름, 닉네임, 전화번호를 조회할 수 있다 |

## 4. 응답 필드 정의

### 4.1 회원 정보 필드

| 항목 | 필드명 | 타입 | 설명 |
|------|--------|------|------|
| 회원 ID | id | Long | 회원 고유 식별자 |
| 이메일 | email | String | 회원 이메일 |
| 이름 | name | String | 회원 실명 |
| 닉네임 | nickname | String | 회원 닉네임 |
| 전화번호 | phone | String | 회원 연락처 |

### 4.2 응답 예시

```json
{
  "content": [
    {
      "id": 1,
      "email": "user@example.com",
      "name": "홍길동",
      "nickname": "길동이",
      "phone": "010-1234-5678"
    }
  ],
  "totalElements": 100,
  "totalPages": 10,
  "number": 0,
  "size": 10
}
```

## 5. 비기능 요구사항

### 5.1 보안

| 구분 | 요구사항 |
|------|----------|
| NFR-01 | 회원 조회 API는 ADMIN 권한만 접근 가능해야 한다 |
| NFR-02 | 인증되지 않은 요청은 401 응답을 반환해야 한다 |
| NFR-03 | ADMIN이 아닌 사용자는 403 응답을 반환해야 한다 |

### 5.2 성능

| 구분 | 요구사항 |
|------|----------|
| NFR-04 | 회원 목록 조회 응답 시간은 500ms 이내여야 한다 |
| NFR-05 | 페이지네이션을 통해 대량 데이터를 효율적으로 처리해야 한다 |

---

## 6. 범위 (Scope)

### 6.1 포함 (In Scope)

- 회원 목록 조회 (페이지네이션)
- 회원 검색 (이름, 닉네임, 이메일)
- 가입일 기준 정렬

### 6.2 제외 (Out of Scope)

- 회원 상세 조회
- 회원 정보 수정
- 회원 승인/거절
- 회원 정지/해제
- 회원 삭제
